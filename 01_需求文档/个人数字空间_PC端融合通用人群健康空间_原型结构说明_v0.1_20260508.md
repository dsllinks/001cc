# 个人数字空间_PC端融合通用人群健康空间_原型结构说明_v0.1_20260508

## 一、文档定位

本文档基于已确认的需求文档 `个人数字空间_PC端融合通用人群健康空间_需求文档_v0.2_20260508.md` 进行原型层级细化，用于指导 Axure / Figma / MasterGo 出图。

- 每个区域承载什么信息

## 二、原型输出范围

本次原型结构说明覆盖 3 类页面：

1. **个人数字空间主界面**
2. **通用人群健康空间详情页**
3. **专项人群健康空间占位页**

不在本次原型范围内的内容：
- 孕产妇/慢病/幼儿/重疾详情页正式内容
- 人群筛选、患者列表、人群管理工作台
- 移动端页面改版

## 三、页面结构总览

### 3.1 页面层级关系

```text
页面1：个人数字空间主界面
  ├─ 点击“通用人群” → 页面2：通用人群健康空间详情页
  ├─ 点击“孕产妇” → 页面3：专项人群占位页
  ├─ 点击“慢病”   → 页面3：专项人群占位页
  ├─ 点击“幼儿”   → 页面3：专项人群占位页
  └─ 点击“重疾”   → 页面3：专项人群占位页
```

### 3.2 原型出图顺序建议

建议先出以下 5 张核心页面/状态图：

1. 主界面默认态
2. 主界面含多个人群标签态
3. 通用人群详情页默认态
4. 通用人群详情页滚动浏览态
5. 专项人群占位页

如时间允许，再补充以下交互状态：
- 诊疗记录详情展开态
- 体征数据溯源弹窗态
- 无数据态
- 搜索筛选后的结果态

## 四、页面1：个人数字空间主界面

### 4.1 页面定位

作为患者个体画像总入口，继续承载：
- 个人基础信息查看
- 标签能力查看
- 人群归属查看
- 进入人群健康空间详情页

### 4.2 页面结构

页面按纵向分为 3 个主区域：

```text
┌────────────────────────────────────────────────────────────┐
│ A. 顶部个人信息区                                           │
├────────────────────────────────────────────────────────────┤
│ B. 标签及人群统计卡区                                       │
├────────────────────────────────────────────────────────────┤
│ C. 个人健康标签图谱区                                       │
└────────────────────────────────────────────────────────────┘
```

### 4.3 A区：顶部个人信息区

#### 作用
展示当前患者的基础身份信息与数据时间信息。

#### 建议字段
- 姓名
- 性别
- 年龄
- 民族
- 证件信息
- 联系方式
- 地址
- 数据截止时间
- 刷新按钮

#### 原型表现建议
- 采用横向信息条或信息卡头部布局
- 左侧展示患者基础身份信息
- 右侧展示数据截止时间与刷新按钮
- 信息区高度不宜过高，避免压缩下方核心区

### 4.4 B区：标签及人群统计卡区

这是本次原型改造重点。

#### B区结构拆分

```text
┌────────────────────────────────────────────────────────────┐
│ 标签统计卡1 │ 标签统计卡2 │ 标签统计卡3 │ 人群统计卡          │
└────────────────────────────────────────────────────────────┘
```

建议保留 4 张卡片并排的布局：
- 卡片1：标签类别
- 卡片2：命中标签数
- 卡片3：高风险标签数
- 卡片4：人群统计卡

#### 人群统计卡内部结构

```text
┌──────────────────────────────┐
│ 标题：归属人群                │
│                              │
│ 通用人群  >                  │
│ 孕产妇    >                  │
│ 慢病      >                  │
│ 幼儿      >                  │
│ 重疾      >                  │
└──────────────────────────────┘
```

实际展示时：
- 只展示患者真实归属的人群
- 所有患者必有“通用人群”
- 专项人群按实际命中情况显示 0~4 个

#### 人群统计卡交互要求

1. 每个人群标签都必须具备可点击感知
2. 标签右侧建议增加箭头或进入图标
3. hover 时需要有高亮反馈
4. 点击行为：
   - 通用人群：进入通用人群健康空间详情页
   - 专项人群：进入专项人群占位页

#### 原型表现建议

**方案建议：使用“纵向标签列表”而不是“横向小标签堆叠”**

原因：
- 横向标签堆叠虽然节省空间，但不利于表达“可进入详情页”的交互含义
- 纵向列表更像导航入口，点击关系更明确
- 后续如果专项人群增多，也更容易扩展

建议状态至少出 2 个：
- 仅通用人群状态
- 通用人群 + 多个专项人群状态

### 4.5 C区：个人健康标签图谱区

#### 作用
继续作为个人标签画像的核心展示区。

#### 保留内容
- 标签图谱
- 标签类别筛选
- 图例说明
- 节点可展开/可下钻状态

#### 原型表现建议
- 整体区块保持现有大结构
- 不需要因为本次改动重做图谱信息结构
- 仅需确保图谱区与上方人群统计卡之间层级清晰

### 4.6 页面1关键交互态

主界面建议至少体现以下状态：
1. 默认态
2. 多个人群归属态
3. hover 到人群入口态
4. 点击通用人群后的跳转态

## 五、页面2：通用人群健康空间详情页

### 5.1 页面定位

承载当前患者在“通用人群健康空间”下的全量健康数据。

该页面不是摘要页，而是**完整详情页**。

### 5.2 页面整体结构

推荐采用：**左侧锚点导航 + 右侧内容分区**

```text
┌──────────────────────────────────────────────────────────────┐
│ A. 页面头部                                                   │
├──────────────┬───────────────────────────────────────────────┤
│ B. 锚点导航区 │ C. 详情内容区                                 │
│              │   C1 诊疗记录                                  │
│              │   C2 健康史                                    │
│              │   C3 体征数据                                  │
│              │   C4 年度账单                                  │
└──────────────┴───────────────────────────────────────────────┘
```

### 5.3 A区：页面头部

#### 作用
建立页面归属、返回路径和患者识别。

#### 建议内容
- 返回按钮 / 面包屑
- 页面标题：通用人群健康空间
- 患者基础信息（姓名、性别、年龄、证件）
- 数据截止时间

#### 原型建议
头部信息比主界面更简洁，重点突出：
- 我现在在哪个空间
- 我看的还是哪位患者
- 如何返回主界面

### 5.4 B区：锚点导航区

#### 作用
作为四大模块的页内定位导航。

#### 锚点项
- 诊疗记录
- 健康史
- 体征数据
- 年度账单

#### 原型建议
- 左侧固定定位
- 当前浏览模块高亮
- 点击后滚动到对应模块首屏
- 导航样式不要设计成 Tab，要明确它是“页内导航”而不是“内容切换”

### 5.5 C区：详情内容区总体原则

1. 四个模块均在同一页面内展示
2. 模块之间采用分区标题 + 留白 + 分隔线区分
3. 每个模块内部允许有自己的二级切换或筛选
4. 一级结构取消 Tab，二级结构保留原移动端模块内部分类逻辑

## 六、C1：诊疗记录模块原型拆解

### 6.1 模块结构

```text
模块标题：诊疗记录
  ├─ 子菜单：全部 / 门诊记录 / 住院记录
  ├─ 搜索筛选区：关键词 + 时间范围
  ├─ 列表区
  └─ 详情展开区（抽屉/侧栏/弹窗）
```

### 6.2 区域说明

#### 子菜单区
- 保留“全部 / 门诊记录 / 住院记录”
- 菜单后带数量
- 放在模块头部下方

#### 搜索筛选区
- 关键词输入框
- 时间范围选择器
- 查询按钮 / 重置按钮

#### 列表区
建议使用 PC 端表格化列表，而不是移动端卡片流：
- 原始诊断名称
- 就诊类型
- 就诊机构
- 就诊科室
- 就诊时间
- 操作（查看详情）

#### 详情区
推荐原型先画“右侧抽屉”方案：
- 不完全打断列表浏览
- 有利于连续查看多条记录
- 适合 PC 端操作习惯

### 6.3 原型状态建议
- 默认列表态
- 搜索筛选态
- 门诊详情展开态
- 住院详情展开态
- 无数据态

## 七、C2：健康史模块原型拆解

### 7.1 模块结构

```text
模块标题：健康史
  ├─ 子菜单：既往病史 / 家族史 / 过敏史 / 个人史
  ├─ 搜索区（前三类展示）
  └─ 内容展示区
```

### 7.2 原型表达重点

#### 既往病史
建议采用表格或纵向列表：
- 首次确诊时间
- 诊断名称
- 诊断机构
- 诊断科室

#### 家族史
建议采用“按亲属分组”的信息块：
- 父亲
- 母亲
- 祖父母/外祖父母等
- 每个亲属块下展示对应病史

#### 过敏史
建议采用“按过敏类型分组”的信息块：
- 药物过敏
- 食物过敏
- 其他过敏

#### 个人史
建议采用“生活方式卡片组”：
- 饮食习惯
- 吸烟情况
- 饮酒情况
- 运动情况
- 摄盐情况

### 7.3 原型状态建议
- 既往病史展示态
- 家族史展示态
- 过敏史展示态
- 个人史展示态
- 节点缺失态
- 全无数据态

## 八、C3：体征数据模块原型拆解

### 8.1 模块结构

```text
模块标题：体征数据
  ├─ 子菜单：血压 / 血糖 / BMI
  ├─ 统计卡区
  ├─ 趋势图区
  └─ 溯源信息弹出层
```

### 8.2 区域说明

#### 子菜单区
- 血压
- 血糖
- BMI

默认选中血压。

#### 统计卡区
建议 4 张小卡横向排布：
- 历年来检查次数
- 历年来异常次数
- 近一年来检查次数
- 近一年来异常次数

#### 趋势图区
- 主图表区域
- 横坐标：日期
- 纵坐标：指标单位
- 下方缩放进度条

#### 溯源弹出层
推荐原型先画“点击数据点后弹出轻量浮层”或“右侧抽屉”两种之一。
若时间有限，优先出轻量弹层。

### 8.3 原型状态建议
- 默认血压态
- 切换血糖态
- 切换BMI态
- 点击数据点溯源态
- 无数据态

## 九、C4：年度账单模块原型拆解

### 9.1 模块结构

```text
模块标题：年度账单
  ├─ 年份切换区
  └─ 消费展示区
```

### 9.2 区域说明

#### 年份切换区
- 年份下拉框 / 左右切换器
- 默认选中当年

#### 消费展示区
建议使用 3 张核心消费卡：
- 门诊消费
- 住院消费
- 体检消费

如空间允许，可增加：
- 消费占比图
- 年度合计金额

但这些属于表现优化，不属于本次必须项。

### 9.3 原型状态建议
- 默认当年态
- 切换年份态
- 无消费数据态

## 十三、原型评审时重点确认的问题

1. 人群统计卡采用“纵向入口列表”还是“横向标签组”
2. 通用人群详情页采用“左侧锚点”还是“顶部锚点”

## 十四、结论

本次原型结构设计的核心不是新增业务，而是把已经确认的融合方案转成更适合 PC 端表达的页面层级：
- 主界面负责入口与总览
- 通用人群详情页负责全量数据承载
- 专项人群先占位，后续补详情

原型绘制时，优先确保“入口清晰、层级清晰、可扩展性清晰”，不要回到 v0.1 的“主页面下方直接堆叠大块内容”的方案。
